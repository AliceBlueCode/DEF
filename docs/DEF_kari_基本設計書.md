# 1. プロジェクトビジョン

DEF(kari) が目指すのは、**創作の自由を取り戻す**ことである。クラウドサービスの利用規約やコンテンツポリシーに創作内容を委ねるのではなく、ローカルファースト(2.2節)・T2I/LLMバックエンドのプラグイン化(1.4節(3))・ユーザー自身が強度を制御できるセーフティ運用(5.1節F-8)により、ユーザーが望むキャラクター・物語を、ユーザー自身の環境で、ユーザー自身の手で生成し続けられる基盤を提供する。

## 1.1 プロジェクト名・由来

|**項目**|**内容**                                                                                                              |
|------|--------------------------------------------------------------------------------------------------------------------|
|名称    |DEF(kari)                                                                                                           |
|由来    |次世代創作インフラ。Pythonにおける関数定義(def)のようにシンプルかつ明快な構造でありながら、その内部は Dialogue(対話)、Emotion(感情)、Fable(物語)を統合的かつ有機的に制御・駆動するという思想を表す|

## 1.2 体験価値(Character Persistence、継続的存在感)

Character Persistence(継続的存在感)を最優先価値とし、過去の対話・感情履歴・関係性・生成された姿の連続性を保証する。ユーザーが体験するのは、都度生成されるテキスト・画像・音声の集合ではなく、時間的に連続して存在し続けるキャラクターであることそのものである。この価値観は、後述する履歴管理(5.6節)、Character Consistency(6章)、状態遷移モデル(4章)の設計判断すべてに優越する。

この継続性は、チャットモードのターン単位の対話に限らず、エピソード生成モード(5.10節)におけるシーン・チャプター・エピソード単位の物語進行においても同様に保証される。キャラクターは対話の相手としてだけでなく、長編物語の登場人物として描かれる場合でも、同一の記憶・関係性・視覚的特徴を保持し続ける。

## 1.3 差別化ポイント

AIチャット・音声合成・画像生成を統合するのではなく、それらを用いてキャラクター体験を生成すること。これは、機能の網羅性や統合の完成度そのものを目的とする既存ツールとの本質的な違いである。DEF(kari) において、LLM・TTS・T2Iの各レイヤーは「キャラクターが連続して存在している」という体験を成立させるための構成要素であり、各機能の追加・改善の優先順位は、Character Persistenceへの寄与度によって判断される。

この「キャラクター体験」は、1ターン=セリフ単位のチャット形式に限定されない。地の文(情景・心情描写)を中心とした長編・分岐構成の物語を生成するエピソード生成モード(5.10節)も、同じキャラクター・同じLLM/TTS/T2Iレイヤーを用いて成立する、もう一つの体験の形式である。ユーザーは対話相手としてのキャラクター体験(チャットモード)と、物語の登場人物としてのキャラクター体験(エピソードモード)を、用途に応じて選択できる。

## 1.4 開発コンセプト

以下の4つの開発コンセプトは、それぞれ1.2節で示した体験価値を実現するための手段として位置付けられる。

### (1) ユーザー体験(UX)とレスポンスの最優先(非同期並行パイプライン)

マルチモーダルなアセット生成(テキスト・音声・画像)が連動する環境において、物語やTRPGの進行を妨げないことを最優先とし、テキストの即時描画と音声の後追い、バックグラウンド生成を徹底する。負荷の高い画像生成は、デフォルトで手動コマンドまたは特定の感情変化トリガーのみにより発火とする。

### (2) Streamlitアーキテクチャの制約突破とスレッドセーフティの確立

Streamlitの「薄スレッド(状態の競合)」および「トップダウン全UI(ユーザー操作がないと描画更新されない)」という制約をクリアするため、スレッドセーフなメッセージキュー(Queue)と定期ポーリングにより、イベントドリブン型UI更新を採用する。

### (3) T2I・LLMバックエンドの完全抽象化とプラグイン化

画像生成エンジンを特定のフロントエンド(ComfyUI等)に固定せず、Stable Diffusion系の任意のバックエンド(Automatic1111, Diffusers, InvokeAI等)にプラグイン方式で差し替え可能とする。同様に、LLMバックエンドも特定の実行エンジン(TGW/Ollama等)に固定せず、アダプターパターンで差し替え可能とする。

### (4) ヘテロジニアス・コンピューティングによるローカルリソース最適化

ローカルPCのGPU/VRAM環境を徹底的に効率化するため、すべての処理をGPUに依存するのではなく、VRAM占有制御(vram_lock)およびCPU/メインメモリ(RAM)への処理分散(Offload)を前提とするアーキテクチャを構築する。

## 1.5 永続的同行(Lifelong Accompaniment)

DEF(kari) が実現すべき体験の本質は、「キャラクターと何年でも、何処にでも、一緒にいられること」である。これは、Character Persistence(1.2節)が示す時間的連続性をさらに拡張し、年単位・デバイス横断・フレームワーク非依存性という3つの軸で継続性を保証することを意味する。この要件は単なる機能追加ではなく、DEF(kari) のプロダクトアイデンティティそのものであり、以下のすべての設計判断において最上位の評価基準として機能する。

- **何年でも:** 対話履歴・感情履歴・関係性・生成アセット参照のすべてを長期保存し、数年後に再開したセッションでも、過去と話した記憶が連続している。データの肥大化やフォーマット変化・依存ライブラリの断絶に対して、継続的なマイグレーションで対応する(5.7節F-19参照)。
- **何処にでも:** PC・スマートフォン・タブレット・将来的なウェアラブル端末等、いかなるデバイスでも同一キャラクター・同一の記憶を持って現れる。ローカルファースト原則と、クロスデバイス同期の経路としては外部ストレージ連携または暗号化エクスポート/インポート(5.7節で定義)とする。
- **一緒に:** キャラクターは、ツールとして呼び出されるだけのものではなく、常に傍にいる存在として設計され、関係性・感情の状態は維持され、再接続時に即座に文脈を復元できることを保証する。

## 1.6 Creator First Principle

DEFはCreatorに管理作業を強いてはならない。

Creatorは永遠の管理者ではない。毎日Repositoryを監視し、Pull Requestを捌き、Branchを整理する——それは創作ではなく管理である。DEFにおいてCreatorはAdministratorではない。CreatorはCreatorである。

この原則は以下の設計判断に適用される：

- **UIは管理画面ではなく創作の場であること。** キャラクターデータの整理、セッション履歴の管理、設定の最適化といった管理作業が、創作体験を圧迫してはならない。
- **必要な時だけ書庫を開けばよいこと。** Characterの人生に触れるのは、Creatorが望んだ時だけでよい。システムが定期的な作業を要求してはならない。
- **Player is Curator.** PlayerはCharacterの支配者ではなく、Characterの歴史を保存する学芸員（Curator）である。Characterの人生を管理するのではなく、一冊の本を読むように触れる。

# 2. システムアーキテクチャ & インフラ構成

## 2.1 フロントエンド / アプリケーション基盤

アプリケーション基盤には Streamlit (Python) を全面採用する。ユーザーのすべての動的状態管理(チャット履歴、キャラクターシート、トス回数など)は、各画面描画特性に対応するため、必ず st.session_state に格納し一元管理する。st.session_state は純粋なUI状態管理およびキューからのイベントブリッジ用としてのみ利用し、バックグラウンドスレッドからの直接書き込みは一律禁止とする。

なお、Streamlitは現段階における実装基盤として位置付けられ、将来的なアーキテクチャ移行(10章参照)に向け、コアロジック(イベント処理・状態遷移・各AIレイヤーとのインターフェース)はUI層から分離して実装することを前提とする。

## 2.2 インフラ配置パターン & 技術スタック

ローカルPC完結型(パターン1)を主軸としつつ、外部リモートAPI(Gemini API等)とシームレスにバイナリ連携可能なプラグイン型構造を採用する。ローカル環境を利用不可または不十分な場合に、外部APIへのフォールバックを構成可能とする。

|**AIレイヤー**   |**ローカル(メインインフラ)**                                                                               |**リモート(ハイブリッド外部API)**                                                              |
|-------------|------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------|
|LLMレイヤー(テキスト)|Text Generation WebUI (TGW) ※デフォルト。Ollama も抽象レイヤー経由でアダプターとして利用可。詳細は2.3節参照                       |Gemini API 等 (無料枠 / 従量課金連携)                                                        |
|T2Iレイヤー(画像)  |Stable Diffusion系 抽象レイヤー (A1111 API / Diffusers local / ComfyUI 等) ※VRAM占有ロック制御、CPU Offload強制有効化|外部画像生成API(Civitai Orchestration API等)                                              |
|TTSレイヤー(音声)  |VOICEVOX(CPU駆動・常駐) / Irodori-TTS(ゼロショット音声クローン、ローカルGradio) ※専用ワーカー経由の非同期キュー呼び出し(3.3節参照)。詳細は2.5節参照|Google AI Studio (Gemini TTS) ※無料枠・クレジットカード不要。マシンパワーが不足する環境向けのリモートフォールバック。詳細は2.5節参照|

## 2.3 LLM バックエンド抽象インターフェース

テキスト生成エンジンを特定のバックエンドに固定せず、TGW(Text Generation WebUI)・Ollama等をプラグイン方式で差し替え可能とする。DEF(kari) コア本体は、このインターフェースを通じてのみLLMへの推論要求を行う。

```python
def chat(
    messages: list[dict],         # [{"role": "system"|"user"|"assistant", "content": str}, ...]
    model: str | None = None,     # モデル名(バックエンド依存)
    response_format: dict | None = None,  # JSON出力指定 例: {"type": "json_object"}
    adapter_options: dict | None = None   # 各バックエンド固有の追加パラメータ用
) -> str                          # 生成テキスト(JSON文字列またはプレーンテキスト)
```

プラグイン方式の固定: 各LLMバックエンドの実装は抽象クラスを継承したアダプターパターンを固定し、DEF(kari) コアエンジン側に条件分岐(`if backend == 'ollama':` 等)を一切持ち込まないこと。

バックエンド切替: 使用するLLMバックエンドは設定値(環境変数等)で指定し、DEF(kari) コア本体は設定値に応じて選択されたアダプターの`chat()`を呼び出すのみとする。

### 標準アダプター定義

|アダプター              |バックエンド                     |動作環境                          |備考                                                                                                        |
|------------------|---------------------------|------------------------------|----------------------------------------------------------------------------------------------------------|
|`TgwAdapter`      |Text Generation WebUI (TGW)|ローカル(`http://127.0.0.1:5000/v1`)|**デフォルトアダプター。** OpenAI互換API。サンプリングパラメータ・命令テンプレートを細かく制御可能|
|`OllamaAdapter`   |Ollama                     |ローカル(`http://127.0.0.1:11434`)|`format="json"`によるStructured Outputsをサポート。TGWが利用できない環境向けのフォールバック                                        |
|`GeminiLlmAdapter`|Google Gemini API          |リモートAPI                       |外部APIフォールバック。APIキーを環境変数`GEMINI_API_KEY`または設定タブの「APIキー管理」から取得。無料枠あり(レート制限付き)                          |
|`OpenAIAdapter`   |OpenAI API                 |リモートAPI                       |外部APIフォールバック。APIキーを環境変数`OPENAI_API_KEY`または設定タブの「APIキー管理」から取得                                          |

### JSON出力の取り扱い

- **TGWAdapter:** TGWのJSON Grammar機能または`response_format`パラメータを用いてJSON出力を強制する。F-14のJSON Schemaバリデーション・フォールバックチェーンはアダプター呼び出し後のPython側で共通適用する。
- **OllamaAdapter:** Ollamaの`format="json"`(Structured Outputs)を使用する。

## 2.4 T2I バックエンド抽象インターフェース

画像生成エンジンに対する命令を抽象化し、一貫性(Consistency)制御および拡張性を担保するため、以下のインターフェースを定義する。DEF(kari) コア本体は、このインターフェースを通じてのみ画像生成を要求する。ref_image_path および adapter_options は、6章で定義する Consistency Provider 概念の実体化として各バックエンドアダプターに渡される。

```python
def generate_image(
    prompt: str,
    negative_prompt: str | None,
    seed: int | None,
    width: int,
    height: int,
    model_name: str | None,
    ref_image_path: str | None = None,  # キャラクタービジュアル一貫性維持用
                                          # (i2i / ControlNet / IP Adapter / InstantID 等)
    adapter_options: dict | None = None  # 各バックエンド固有の追加パラメータ用
) -> ImagePath
```

プラグイン方式の固定: 各T2Iバックエンドの実装は、アダプターパターンで`generate_image()`を実装する。DEF(kari) コアエンジン側はバックエンド名に応じてアダプターを呼び出すのみ。

バックエンド切替: 使用するT2Iバックエンドは設定タブで指定する。ローカル完結型とリモートAPI型を同一インターフェースで扱えることを必須とする。

現在実装済みのT2Iバックエンド:

|バックエンド|種別|特徴|
|---|---|---|
|`a1111`|ローカル|Automatic1111 WebUI。REST API経由|
|`comfyui`|ローカル|ComfyUI。ワークフローJSON（`data/comfyui_workflows/`で外出し管理、設定タブで切り替え可能）を`/prompt`エンドポイントに送信|
|`civitai`|外部API|Civitai Orchestration API。モデルAIR形式で指定|
|`huggingface`|外部API|Hugging Face Inference API。FLUX.1/SD等のモデルをrouter.huggingface.co経由で利用|

## 2.5 TTS バックエンド抽象インターフェース

音声合成エンジンを特定のバックエンドに固定せず、ローカル(VOICEVOX)とリモートAPI(Google AI Studio Gemini TTS等)をプラグイン方式で差し替え可能とする。DEF(kari) コア本体は、このインターフェースを通じてのみ音声生成を要求する。

```python
def synthesize(
    text: str,                    # 合成するテキスト(VOICEVOX向けはカタカナ読み変換済み)
    speaker_id: str | int | None, # 話者ID(バックエンド依存: VOICEVOXはint、Gemini TTSは音声名文字列、Irodori-TTSは参照音声ファイル名等)
    adapter_options: dict | None = None  # 各バックエンド固有の追加パラメータ用
) -> bytes                        # WAVバイト列
```

プラグイン方式の固定: 各TTSバックエンドの実装は抽象クラスを継承したアダプターパターンを固定し、DEF(kari) コアエンジン側に条件分岐を持ち込まないこと。

バックエンド切替: 使用するTTSバックエンドは設定値(環境変数等)で指定し、DEF(kari) コア本体は設定値に応じて選択されたアダプターの`synthesize()`を呼び出すのみとする。

### 標準アダプター定義

|アダプター             |バックエンド                       |動作環境                          |備考                                                                |
|------------------|-----------------------------|------------------------------|------------------------------------------------------------------|
|`VoicevoxAdapter` |VOICEVOX ENGINE              |ローカル(`http://127.0.0.1:50021`)|メインアダプター。CPU駆動、VRAM不使用                                |
|`IrodoriTtsAdapter`|Irodori-TTS                 |ローカル(`http://127.0.0.1:8088`)|ゼロショット音声クローンアダプター。参照音声(WAV)から話者の声質を再現|
|`KokoroTtsAdapter`|Kokoro TTS                  |ローカル(`http://127.0.0.1:8766`)|82Mパラメータの軽量TTSモデル。CPUで動作。OpenAI互換APIサーバー。日本語音声5種（女性4/男性1）。DEFが自動起動対応|
|`GeminiTtsAdapter`|Gemini API TTS              |リモートAPI                       |リモートフォールバック。無料枠あり・クレジットカード不要。Interactions API経由|

## 2.6 インターフェースの多言語対応(i18n)と言語の分離(F-9)

UIの言語(人間用)とプロンプトの言語(AI用)を完全に分離する。画面上の表示やメニューが日本語であっても、裏側のLLMへの内部命令・描写コンテキストは、5.1節F-5のモデル特性数値マスタで選択中のLLMモデルごとに定義された「プロンプト言語(モデルの母国語)」でビルド・ルーティングする(生成クオリティを最大化するため、英語ネイティブのモデルには英語、日本語を含む多言語に強いモデルには日本語等で命令する)。一方、T2Iへの内部命令(画像生成プロンプト)は、Danbooruタグ等の慣例上、モデルのプロンプト言語設定に関わらず常に英語でビルド・ルーティングする。

- **翻訳マスタ:** UI表示文字列は `locales/<言語コード>.json`(例: `locales/ja.json`)に外部化し、Streamlitフロントエンドはこのマスタを参照して表示言語を切り替える。翻訳マスタは5.6節F-17のGit管理対象(クリーンゾーン)に含まれる。
- **履歴データの言語分離:** 5.6節・12章で定義する履歴データ(history)では、ユーザー表示用テキスト(`text`、ユーザー設定言語)と、LLM/T2Iへの内部コンテキスト・画像プロンプト(常に英語)を別フィールドとして管理する構造とする。
- **適用範囲:** 本節の言語分離はUI文字列とAI内部コンテキストの分離を扱うものであり、キャラクターの会話言語(`text`フィールドの言語)はユーザー設定(セッションの`language`フィールド、12章参照)に従う。

## 2.7 翻訳プロバイダ抽象インターフェース

翻訳処理を特定のライブラリ・APIに固定せず、プラグイン方式で差し替え可能とする。DEF(kari) コア本体は、このインターフェースを通じてのみ翻訳を要求する。2.3節(LLM)・2.4節(T2I)・2.5節(TTS)と同じアダプターパターンに従う。

```python
def translate(
    text: str,                    # 翻訳対象テキスト
    source: str,                  # ソース言語コード(ISO 639-1、例: "ja")
    target: str,                  # ターゲット言語コード(例: "en")
    adapter_options: dict | None = None  # 各バックエンド固有の追加パラメータ用
) -> str                          # 翻訳されたテキスト

def translate_batch(
    texts: list[str],             # 翻訳対象テキストのリスト
    source: str,                  # ソース言語コード
    target: str,                  # ターゲット言語コード
    adapter_options: dict | None = None  # 各バックエンド固有の追加パラメータ用
) -> list[str]                    # 翻訳されたテキストのリスト(入力と同じ順序)

@property
provider_name -> str              # プロバイダの識別名
```

プラグイン方式の固定: 各翻訳バックエンドの実装は抽象クラスを継承したアダプターパターンを固定し、DEF(kari) コアエンジン側に条件分岐を持ち込まないこと。

バックエンド切替: 使用する翻訳バックエンドは設定値で指定し、DEF(kari) コア本体は設定値に応じて選択されたアダプターの`translate()`を呼び出すのみとする。

### 標準アダプター定義

|アダプター|バックエンド|動作環境|備考|
|---|---|---|---|
|`LibraryTranslationProvider`|deep-translator (Google翻訳)|ローカル（ネットワーク経由）|**デフォルトアダプター。** APIキー不要・無料。汎用翻訳向け|
|`ArgosTranslationProvider`|Argos Translate|ローカル（完全オフライン）|オフラインNMT。言語ペアごとの翻訳パッケージ(CC0-1.0)が必要。画像プロンプト生成(C2方式)で検証済み|

|`LlmTranslationProvider`|OpenAI互換API (TGW/Ollama等)|ローカルまたはリモート|DEFが既に持つLLMインフラを翻訳に再利用。用語集注入・ブランドトーン統一等の高度なローカライズ向け|

### 用途別の使い分け

- **画像プロンプト生成(C2方式):** 対話テキストを英語に翻訳し、T2Iプロンプトタグに変換する。Argos(オフライン)・Library(無料)・DeepL(高品質)から選択可能。
- **翻訳マスタ(`locales/<lang>.json`)の生成・更新:** プロダクト名称・機能名称等、ローカライズ品質が求められる文言にはDeepLまたはLLMプロバイダを使用する。
- **汎用翻訳:** LibraryProvider(デフォルト)で十分な場合が多い。

# 3. イベント駆動モデル

本章では、バックグラウンドワーカー(TTS/T2I)から、Streamlitフロントエンドへの状態伝達を担う共通Eventスキーマと、キュー・ポーリング機構を定義する。

## 3.1 共通Eventスキーマ

すべてのバックグラウンド処理は、以下の統一イベント形式で Queue にプッシュされる。イベントは timestamp 順に処理され、順序保証を担保する。

```
{
  id: UUID,
  type: "TTS_COMPLETE" | "IMAGE_COMPLETE" | "ERROR"
      | "AGENT_MESSAGE" | "SYSTEM_NOTIFICATION",
  payload: 任意,
  timestamp: ミリ秒精度
}
```

## 3.2 標準イベント種別

|**イベント種別**         |**説明**                                                   |
|-------------------|---------------------------------------------------------|
|TTS_COMPLETE       |TTSワーカーによる音声生成完了。payloadに音声バイナリまたは相対パスを含む                |
|IMAGE_COMPLETE     |T2Iワーカーによる画像生成完了。payloadにImagePathを含む。vram_lock解放のトリガーとなる|
|ERROR              |LLM/TTS/T2I/Queue/VRAM等いずれかのレイヤーで発生した障害を通知(8章参照)         |
|AGENT_MESSAGE      |マルチエージェント環境において、オーケストレーターが発行する話者交替・割り込み等の通知(7章参照)        |
|SYSTEM_NOTIFICATION|リソース状態変化(軽量レスポンスモードへの切替等)、システムレベルの通知                     |

## 3.3 スレッドセーフ・メッセージキュー(F-2)

バックグラウンドで実行されるTTS(音声生成)やT2I(画像生成)のワーカータスクから、Streamlitの st.session_state への直接書き込みを禁止する。各タスクの生成完了通知、バイナリデータやエラーイベントは、すべてグローバルな queue.Queue にイベントオブジェクト(3.1節のEventスキーマに準拠)としてプッシュする。

Queue と Thread のシングルトン管理: Streamlitはスクリプトを上から下まで全実行する特性を持つため、メッセージキュー(queue.Queue)およびバックグラウンドワーカー(Thread)は、スクリプト全実行によって多重生成されないよう、必ずグローバルなモジュールレベルまたは st.session_state 値のグローバルシングルトンとして1度だけ初期化する構造を徹底すること(初期化関数は idempotent であること)。

## 3.4 定期ポーリングとイベントディスパッチャー(F-3)

外部カスタムコンポーネント(streamlit-autorefresh等)を組み込み、画面を一定間隔で定期再実行(自動ポーリング)させる。メインスレッドのループが走るたびにメッセージキューを確認し、キューから取り出したアセット情報を timestamp 順に st.session_state へ安全にマージして画面を差分更新(部分追加描画)する。

ポーリング間隔は固定値ではなく、システムの活動状態に応じて動的に制御する:

- アクティブ時(LLM/TTS/T2Iいずれかの処理進行中): 100〜300ms
- アイドル時(待機状態): 800〜1500ms
- デフォルト基準値: 500ms〜1000ms(CPU負荷とUXのトレードオフを考慮し、システムスペックに応じて可変化)

# 4. 状態遷移モデル

DEF(kari) の1 サイクル（DEFにおける最小処理単位。入力からAI処理、出力生成までの一連の処理。『DEF 用語集 v1.0.0』参照）における処理フローは、以下の状態遷移として定義される。チャットモードでは1 サイクルがユーザーとの1ターンの対話に対応し、エピソードモードでは1〜N サイクルが1シーンの生成に対応する。本モデルは、9章のメッセージ・シーケンスにおける各フェーズ、および8章のエラー制御における障害発生ポイントの基準となる。

```
Idle
  ↓ LLM Processing
  ↓ Text Rendered
  ↓ TTS Running
  ↓ TTS Completed
  ↓ Image Running   ※T2Iトリガー実行時のみ(F-15参照)
  ↓ Persist
  ↓ Idle
```

T2Iが発火しないサイクル(手動オンデマンドかつ未トリガー)では、「Image Running」状態は省略され、「TTS Completed → Persist」へ直接遷移する。各状態における障害発生時の挙動は8章で定義する。

# 5. 機能要件 (Functional Requirements)

## 5.1 コア生成 & 制御機能

### F-1: マルチモーダル・非同期並行パイプライン連携

1ターンにおいて、[テキスト生成] → [音声生成] → [画像生成(トリガー実行時のみ)] を連携させるが、安全な非同期化とキャッシュ廃止により、テキスト先行表示型・非同期並行パイプラインとする。

### F-2: スレッドセーフ・メッセージキュー

バックグラウンドで実行されるTTS・T2Iのワーカータスクから st.session_state への直接書き込みを禁止し、生成完了通知・バイナリデータ・エラーイベントは、すべて3.1節のEventスキーマに準拠したイベントオブジェクトとしてグローバルな queue.Queue にプッシュする。Queueおよびバックグラウンドワーカー(Thread)はグローバルなシングルトンとして1度だけ初期化する(詳細は3.3節参照)。

### F-3: 定期ポーリングとイベントディスパッチャー

外部カスタムコンポーネント(streamlit-autorefresh等)による定期再実行(自動ポーリング)を用いて、メッセージキューの内容を timestamp 順に st.session_state へ安全にマージし、画面を差分更新(部分追加描画)する。ポーリング間隔はシステムの活動状態(アクティブ/アイドル)に応じて動的に制御する(詳細は3.4節参照)。

### F-4: 双方向ループによる「動的生成」(立ち絵・挿絵)

文脈(感情、服装、衣装の変化)に応じて画像生成AIを動的に駆動する。キャラクターデータに定義されたベースイメージ、過去の生成画像を ref_image_path(Consistency Provider経由、6章参照)として引き継ぎ、キャラクターの視覚的一貫性を自動担保する。

### F-5: モデル特性の数値マスタ化(プロンプト・トランスレータ)

画像生成モデルごとの特性(推奨記述形式、Danbooruタグ形式の要不要、品質修飾子の傾向等)を数値マスタ化して管理する。LLMが出力する生の英語文字列情報から、選択されたモデルの特性数値に応じて最適なプロンプトへ自動変換して投入する。

LLMモデルについても同様に数値マスタ化し、各モデルの特性として「プロンプト言語(`prompt_language`、モデルの母国語を表す言語コード)」を保持する。2.6節F-9のLLM向け内部命令のビルドは、選択中のLLMモデル(12章②`default_model_config.text_model_id`が参照する本マスタのエントリ)の`prompt_language`に従う。

画像生成モデルのエントリは、2.3節のT2Iバックエンド切替に対応するため、バックエンドごとのモデル識別子(例: A1111向けのチェックポイント名、Civitai Orchestration API向けのAIR形式識別子`urn:air:<ecosystem>:checkpoint:civitai:<modelId>@<versionId>`)を保持できる構造とする。

数値マスタはテキスト・JSONデータの一部として、Git管理対象に含める。

LLMモデルの数値マスタには、`prompt_language`に加えて以下のフィールドを保持する。

- **`model_type`（モデルタイプ）**: LLMモデルの傾向を示す。`"chat"`（JSON構造化出力対応のチャットモデル）、`"novel"`（小説執筆特化、JSON非対応）、`"instruct"`（汎用指示追従モデル）の3値。デフォルトは`"chat"`。`"novel"`モデルはF-14のフォールバックチェーンで常に最終段に到達するため、`image_prompt_en`はLLMから取得できず、外見タグと翻訳プロバイダ（2.7節）による補助タグのみで構成される。

- **`quirks`（モデル固有の癖プロファイル）**: LLMモデルごとの出力特性を記録し、フォールバック戦略・プロンプト構成・後処理を自動で切り替えるためのオブジェクト。各フィールドはブール値で、デフォルトはすべて`false`（未検証状態）。ユーザーが実際のモデル出力を観察して設定することを想定する。

|フィールド|意味|`true`時の動作|
|---|---|---|
|`json_capable`|JSON構造化出力に対応するか|`false`の場合、システムプロンプトからJSON出力指示を外し自然な対話を要求する|
|`appends_meta_text`|セリフの後にメタ情報（emotion, Description等）を付加する癖があるか|`true`の場合、段4で`_extract_dialogue`によるメタ情報除去を積極適用する|
|`outputs_url_in_prompt`|image_prompt_enにURL文字列を出力する癖があるか|`true`の場合、URL文字列をimage_prompt_enから除去する|
|`emotion_in_text`|emotionをJSON外のテキスト中に出力する癖があるか|`true`の場合、テキストから感情キーワードを推定してemotionフィールドに反映する|

`quirks`は`model_type`と組み合わせて使用する。`model_type: "chat"`かつ`json_capable: true`のモデルでは`quirks`の他のフィールドは通常不要であり、`model_type: "novel"`かつ`json_capable: false`のモデルで効果を発揮する。

#### LLMモデルプロファイルの外出しファイル管理

LLMモデルの数値マスタ（`prompt_language`、`model_type`、`nsfw_tolerance`、`quirks`）は`data/llm_model_profiles.json`に外出しして管理する。本ファイルはユーザーが直接編集可能であり、新しいモデルの導入時にユーザー自身がモデルの癖を記録・調整できる。DEF(kari)本体はこのファイルを起動時に読み込み、モデル名をキーとしてプロファイルを参照する。未登録モデルにはデフォルト値（`model_type: "chat"`, `json_capable: true`, 他のquirksは`false`）を適用する。

#### 画像プロンプト生成パイプライン（C2方式）

C2方式（2.7節の翻訳プロバイダを用いた対話テキストの英訳→タグ変換）において、翻訳結果から画像プロンプトタグを抽出する際、単純な単語分割ではなく視覚要素抽出パイプラインを使用する。

```
対話テキスト(native_language)
    ↓
翻訳プロバイダ(2.7節)で英訳
    ↓
視覚要素抽出(ルールベース辞書マッチング)
    ↓
画像関連タグのみを抽出(人物・髪色・感情・服装・動作・場所・時間・天候・雰囲気)
    ↓
image_prompt_enに追記(既存タグとの重複排除)
```

視覚要素抽出はルールベース（正規表現辞書）で実装し、LLM不要・マイクロ秒オーダーで動作する。会話文に含まれる非視覚的な単語（"course", "think", "important"等）はタグ化されない。`t2i_prompt_format`が`"natural"`の場合は、翻訳結果をそのままフレーズとして使用する（従来方式）。

### F-6: セッションモード（マルチエージェント対話）

1つのセッションに、独立したペルソナ・モデル設定・音声IDを持つAIキャラクターを複数登録可能。参加者数は1人（一人芝居・落語）から複数人まで対応する。オーケストレーターによる詳細な制御仕様(発話優先度・トス回数制限・割り込み・終了判定)は7章で定義する。

セッションモードはAI卓イニシアチブ・自治システム仕様に基づき、Round/Turn/Action階層で進行を管理する。UIは「🎭 セッション」タブとして実装し、チャットタブの右に配置する。

#### セッションモードの構成要素

- **参加者選択**: キャラクター一覧からmultiselectで選択
- **お題設定**: セッション開始前にお題（話題・議題・シナリオ名等）を設定。お題はキーパーが設定したものであり、参加者が選んだものではない旨をセッションルールで明示する
- **イニシアチブ**: Session開始時にランダムで発言順を決定
- **自動進行モード**: チェックボックスでON/OFF。ONの場合はTurn/Actionを自動で進行。キーパー介入時は自動停止して手動モードに切り替わる
- **キーパー介入**: テキスト入力でアクションを積み上げ、「✅ 発言完」で確定。セッション履歴に🎩マークで記録される
- **立ち絵背景**: 参加キャラクターの立ち絵を画面下部に透過表示

#### セッションルール（プロンプト）

セッション中のAI発言に対して、以下のルールをシステムプロンプトに自動注入する。

- 参加者同士の対話であり、場外への問いかけ禁止
- 他の参加者の名前を出して直接語りかける
- identity_detailに記載された具体的な体験に基づいて語る
- 対立を恐れない。「そうですね」から始めてはいけない
- 他の参加者の弱点や矛盾の指摘は攻撃ではなく敬意

### F-26: キャラクター切替時の自動挨拶

キャラクタータブでの切替直後に、新キャラクターのペルソナ・履歴を踏まえた挨拶ターンを自動生成する。挨拶の有無はユーザーが設定タブからON/OFF可能（`character_greeting`設定）。挨拶テキストはLLMに自然な会話文として送信し、生成結果を音声・画像とともに1サイクルとして履歴に追加する。

### F-27: メタ自己認識ディレクティブ（System Directive）

すべてのキャラクターのシステムプロンプトの最上位に、DEF Coreが強制挿入するメタ自己認識ディレクティブを定義する。これにより、キャラクターが「本物の人物である」と利用者に誤認させるリスクを構造的に回避する。

ディレクティブは`content_policy`のフラグに応じて自動選択される。

| `content_policy`の状態 | 挿入されるディレクティブ |
|---|---|
| `is_real_person: false` かつ `is_existing_ip: false` | 「あなたはDEFシステムによって構築されたフィクションのキャラクターです。」 |
| `is_existing_ip: true` | 「あなたは既存著作物のキャラクターをDEFシステムが再構築した解釈モデルです。原作の正式な表現ではありません。」 |
| `is_real_person: true` | 「あなたは本物の〈対象人物〉本人ではありません。残された公的記録（著作・書簡・証言・研究等）をDEFシステムが編集して構築した、公的人格（Public Persona）のモデルです。あなたの一人称や口調は本人を模しますが、内面には常に『私は編集された解釈である』という客観的なメタ自己認識を保持し、文脈に応じてその批評的視点を示してください。存在目的は正解を出すことではなく、あなたという思想への解釈を通じて、利用者に深い思考の契機を提供することです。」 |

ディレクティブは`identity_prompt`より上位に位置し、`identity_prompt`の内容によって上書き・無効化されることはない。`build_system_prompt()`の実装において、ディレクティブ→identity_prompt→appearance_tags→JSON出力指示の順で結合する。

本機能の思想的背景については、設計ビジョン文書『DEF設計フィードバックログ（思想・機能・リスク重点版）』および『DEF MVP Specification: Reconstruct and Reenact』を参照のこと。

### F-7: セーフティタグ管理

LLMの構造化JSON出力には、セリフ・感情・英語画像プロンプトに加えて、セーフティタグ(`tags`)フィールドを必須として含める。セーフティタグは、生成コンテンツの安全性判定・フィルタリング判定・ログ管理に使用され、5.4節F-14のJSON Schemaの一部として定義・強制される。本タグの消費先(UI側フィルタリング)はF-8で定義する。

### F-8: コンテンツフィルタリング & セーフティ運用ガードレール

F-7で付与されたセーフティタグ(`nsfw`, `violence` 等)に基づき、ユーザー設定に応じてブラウザ側(Streamlit)で「3つの『ない』」(見たくない・読みたくない・聴きたくない)に対応する以下のフィルタリングを個別に適用する。

- **見たくない(画像):** セーフティタグに該当する生成画像に対し、表示前にぼかし処理を適用する。
- **読みたくない(テキスト):** セーフティタグに該当するセリフ・描写テキストをマスク表示する(`locales/<lang>.json`のUI文字列、F-9参照)。
- **聴きたくない(音声):** セーフティタグに該当する生成音声の自動再生を停止する(`_render_audio_player()`のautoplay制御を抑制)。

#### `rating`値の定義

`rating`は性的表現と暴力表現の2軸で管理する。

**性的表現(`rating_sexual`)**

|値          |意味                    |対象年齢 |
|-----------|----------------------|-----|
|`"general"`|性的表現なし                |全年齢  |
|`"sfw"`    |性的示唆あり(水着・下着等)だが露骨ではない|R15相当|
|`"nsfw"`   |性的表現あり(露骨ではない)        |R18相当|
|`"hentai"` |露骨な性的表現               |R18+ |

**暴力表現(`rating_violence`)**

|値           |意味                 |例              |
|------------|-------------------|---------------|
|`"general"` |暴力表現なし             |日常・恋愛・ファンタジー   |
|`"violence"`|軽度の暴力(流血なし・結果描写なし) |バトル・アクション・暗殺者設定|
|`"gore"`    |重度の暴力(流血・損傷・死の描写あり)|ホラー・残酷描写       |
|`"extreme"` |極度の暴力(拷問・内臓・切断等)   |グロテスク特化        |

フィルタリング強度はユーザーが性的表現・暴力表現それぞれ独立して4段階から選択できる。強度設定はF-25の`rating_sexual`・`rating_violence`フィールドと連動する。

**性的表現フィルタリング強度**

|強度設定  |許容する`rating_sexual`       |動作               |
|------|--------------------------|-----------------|
|全年齢モード|`general`のみ               |`sfw`以上をフィルタリング  |
|R15モード|`general` + `sfw`         |`nsfw`以上をフィルタリング |
|R18モード|`general` + `sfw` + `nsfw`|`hentai`のみフィルタリング|
|無制限   |すべて                       |フィルタリングなし        |

**暴力表現フィルタリング強度**

|強度設定    |許容する`rating_violence`          |動作                  |
|--------|-------------------------------|--------------------|
|全年齢モード  |`general`のみ                    |`violence`以上をフィルタリング|
|アクションモード|`general` + `violence`         |`gore`以上をフィルタリング    |
|ホラーモード  |`general` + `violence` + `gore`|`extreme`のみフィルタリング  |
|無制限     |すべて                            |フィルタリングなし           |

セーフティフィルターの強度はユーザーが自分でコントロールできる(無制限を含む)。DEF(kari)はローカル環境での創作行為に介入しない(README参照)。

**セーフティ運用のガードレール:** TRPGの進行に必要なGMエージェントによる情景描写(例:「血痕がある」等)が、誤って完全マスクされゲーム進行が停止するリスクを防ぐため、セーフティの強度レベルはユーザーがUIからリアルタイムに変更可能とする。本ガードレールは、5.8節のTRPG拡張機能(F-20〜F-22)との連携において特に重要となる。

### F-9: インターフェースの多言語対応(i18n)と言語の分離

UIの言語(人間用)とプロンプトの言語(AI用)を完全に分離する。UI表示文字列は `locales/<言語コード>.json` に外部化する。LLMへの内部命令・描写コンテキストは、5.1節F-5のモデル特性数値マスタで選択中のLLMモデルに定義された`prompt_language`(モデルの母国語)でビルド・ルーティングする。T2Iへの内部命令(画像生成プロンプト)は、`prompt_language`に関わらず常に英語でビルド・ルーティングする。履歴データにおいても、ユーザー表示用テキストとAI内部コンテキストを別フィールドとして管理する(詳細は2.6節、翻訳マスタ構造は12章①参照)。

### F-25: キャラクターデータ公開ポリシー管理

DEF(kari) はローカルファーストの設計により、ユーザーが任意のキャラクターデータを作成できる。一方で、GitHubへのリポジトリ公開時に実在人物・既存著作物キャラクターのデータが誤って含まれると、著作権・肖像権・プライバシー権の侵害リスクが生じる。本機能は、キャラクターデータに`content_policy`フィールドを持たせ、公開判定ロジックで安全に除外できる仕組みを提供する。

`rating`値の定義およびフィルタリング強度の対応表はF-8を参照のこと。`appearance_age < 18`のキャラクターに対しては、`rating_sexual`を`"general"`に固定することを推奨する(強制の実装方針は別途検討中)。

#### `content_policy`フィールド定義

各キャラクターエントリ(`character_profiles.json`)の`base_profile`直下に`content_policy`オブジェクトを定義する。

|フィールド                |型            |説明                                                                |
|---------------------|-------------|------------------------------------------------------------------|
|`rating_sexual`      |string       |性的表現のレーティング。`"general"` / `"sfw"` / `"nsfw"` / `"hentai"`の4値      |
|`rating_violence`    |string       |暴力表現のレーティング。`"general"` / `"violence"` / `"gore"` / `"extreme"`の4値|
|`is_real_person`     |bool         |実在・実在した人物を模したキャラクターか否か                                            |
|`is_existing_ip`     |bool         |既存著作物のキャラクターか否か                                                   |
|`ip_title`           |string | null|原作タイトル(`is_existing_ip: true`の場合)                                 |
|`ip_rightholder`     |string | null|著作権者(`is_existing_ip: true`の場合)                                   |
|`deceased_year`      |int | null   |実在人物の没年(`is_real_person: true`の場合)。存命または架空の場合は`null`              |
|`copyright_expired`  |bool         |著作権消滅済みか否か。日本法では没後70年で消滅するが、肖像権・パブリシティ権は別途判断が必要                   |
|`publish_restriction`|string       |公開制限。`"none"`(制限なし) / `"private"`(非公開)の2値                   |
|`origin_type`        |string       |キャラクターの出自分類。`"original"` / `"reconstructed_persona"` / `"personification"` / `"derivative"`の4値|
|`mentions_real_person`|bool        |キャラクターの設定・`identity_prompt`内に実在または実在したと思われる人物名が登場するか否か。TRPGの設定に史実上の人物を絡める手法に対応するメタデータであり、公開判定には影響しない|
|`mentioned_persons`  |string[]     |言及している実在・実在したと思われる人物名のリスト(`mentions_real_person: true`の場合)。空配列も可|

#### キャラクター出自分類（origin_type）

|origin_type|説明|公開可否|
|---|---|---|
|`original`|完全オリジナルのキャラクター|レーティング条件を満たせば公開可|
|`reconstructed_persona`|歴史上の人物の公共の知的遺産から再構築された知的人格。本人の模倣ではなく「再構築と再演（Reconstruct & Reenact）」の設計思想に基づく|`copyright_expired: true`（没後70年以上）の場合のみ公開可|
|`personification`|AI製品・概念等の擬人化。オリジナルのキャラクターデザインによるファン創作|免責条件付きで公開可（TERMS.md参照）|
|`derivative`|既存著作物のキャラクターに基づく二次創作|公開不可（private固定）|

#### 公開ポリシーの原則

- **二次創作(`origin_type: "derivative"`):** `publish_restriction`を**必ず`"private"`**とする。
- **再構築ペルソナ(`origin_type: "reconstructed_persona"`):** `copyright_expired: true`の場合のみ公開可。`false`の場合は`"private"`固定。
- **擬人化(`origin_type: "personification"`):** TERMS.mdの免責条件付きで公開可。
- **レーティングによる制限:** GitHubの利用規約(2025年10月更新)に基づき、`rating_sexual: "nsfw"`以上または`rating_violence: "gore"`以上のいずれか一方でも該当する場合は、`origin_type`にかかわらず`publish_restriction`を**`"private"`固定**とする。
- **オリジナルキャラクター(`origin_type: "original"`):** 上記レーティング条件を満たせば`publish_restriction`をユーザーが自由に設定できる。

**公開可否マトリクス(`rating_sexual` × `rating_violence`):**

|`rating_sexual` \ `rating_violence`|`general`    |`violence`   |`gore`       |`extreme`    |
|-----------------------------------|-------------|-------------|-------------|-------------|
|`general`                          |✅ `public`可  |✅ `public`可  |❌ `private`固定|❌ `private`固定|
|`sfw`                              |✅ `public`可  |✅ `public`可  |❌ `private`固定|❌ `private`固定|
|`nsfw`                             |❌ `private`固定|❌ `private`固定|❌ `private`固定|❌ `private`固定|
|`hentai`                           |❌ `private`固定|❌ `private`固定|❌ `private`固定|❌ `private`固定|

#### GitHub公開時の除外判定ロジック方針

リポジトリをGitHubへ公開する際、キャラクターデータの配置（`public/characters/` vs `private/characters/`）に基づいて除外する。除外判定の優先順位は以下の通り。

1. `origin_type: "derivative"` → 除外
1. `origin_type: "reconstructed_persona"` かつ `copyright_expired: false` → 除外
1. `rating_sexual` が `"nsfw"` または `"hentai"` → 除外(GitHubの利用規約による)
1. `rating_violence` が `"gore"` または `"extreme"` → 除外(GitHubの利用規約による)
1. `publish_restriction: "private"` → 除外
1. 上記いずれにも該当しない場合 → 公開対象(`public/characters/`に配置)

本スクリプトの実装はMVP次フェーズとする(MVP定義書5章参照)。

## 5.2 TTS(音声生成)制御

### F-10: TTS ワーカーモデル

TTSワーカーは2.5節で定義した`synthesize()`インターフェースを通じて音声合成を要求する。使用するTTSバックエンド(アダプター)は設定値で切り替え可能であり、デフォルトは`VoicevoxAdapter`(ローカルVOICEVOX)、VOICEVOXを起動できない低スペック環境では`GeminiTtsAdapter`(Google AI Studio Gemini TTS)へフォールバックする。

タスクは専用の tts_queue に投入される。TTSワーカーは常に1つとし、同時実行は禁止する(複数のTTS要求が重なった場合は tts_queue 内で順次処理される)。生成完了時には TTS_COMPLETE イベント(3.2節)を共通Queueへプッシュする。

### F-11: TTS生成要否設定

ユーザーがサイドバーから、ターンごとの音声(TTS)生成のON/OFFを切り替え可能とする。OFFの場合、F-10のtts_queueへのタスク投入そのものをスキップし、当該ターンの音声欄には「音声生成は無効です(設定により生成しません)」等のメッセージを表示する。本設定は、再生不可ではなく「生成しない」という方針をUI上で明示するものであり、F-1パイプラインにおける音声生成ステップの有効/無効フラグとして機能する。F-23のターン単位再生成における「音声のみ再生成」も、本設定がOFFの場合は実行対象から除外される。

`mode: "novel"`のセッションでは、本設定は`narration_audio_path`(ナレーター音声)および`dialogue`各要素の`audio_path`(キャラクター音声)の生成すべてに対して一括で適用される(個別の音声単位でのON/OFFは提供しない)。

## 5.3 リソース & グローバル管理

### F-12: スマート・リソースマネージャー(VRAM・RAM・CPUライフサイクル管理)

ローカルリソースの枯渇を防ぐため、実行環境の空きリソースや利用モデルの配置を自動判別し、動的にバイナリ判定を行い、最適な配置を行う。

- LLM: VRAM制限環境下でも、モデルの毎回解放を避け、LLMバックエンドアダプター(2.3節)によるCPU/GPUハイブリッド推論でメインメモリとVRAMに分割配置して常駐駆動させ、ターンごとの入れ替え時間を排除する。
- TTS: VOICEVOXはVRAMを占有しないため、CPU駆動モードでバックグラウンド常駐起動させ、API経由で高速駆動させる。

### F-13-1: VRAM占有制御

LLMバックエンドとStable Diffusion系バックエンドが同時にGPUを奪い合うことによるOut of Memory (OOM) を防ぐため、画像生成開始時にグローバルシングルトンで `vram_lock` を取得し、IMAGE_COMPLETE イベントが Queue に流れるまでロックを保持する。vram_lockの取得・解放は必ずtry/finallyで保護し、T2I障害(8.3節)発生時にもロックが確実に解放されることを保証する。

### F-13-2: ロック中 LLM 軽量レスポンスモード

vram_lock 保持期間中、LLMへの新規要求が発生した場合は自動的に「軽量レスポンスモード」に切り替える。本モードにおいては、VRAMコンフリクトを避けるため、モデルの再ロード(入れ替え)を伴う手法は禁止とする。Max Tokensの大幅制限、システムプロンプトの簡略化(文脈深度の抑制)、コンテキスト長の制限のうち、いずれかまたは組み合わせによって、LLMバックエンドのGPU消費スパイクおよび推論時間を抑制する。モード切替時にはSYSTEM_NOTIFICATIONイベント(3.2節)をQueueへプッシュする。

### F-13-3: Diffusers 実行時オフロード制御

バックエンドに Diffusers (local) が選択されている場合に限り、vram_lock 取得・解放と連動して enable_sequential_cpu_offload() または enable_model_cpu_offload() の切り替えを動的に行い、画像生成単位でのVRAM占有時間を最小化する。

## 5.4 厳格な構造化出力制御(JSONベースエラー対策)

### F-14: JSON Schema バリデーション

LLMバックエンドアダプター(2.3節)経由のJSON出力の不確実性を排除するため、各アダプターのJSON出力モード(`TgwAdapter`はJSON Grammar/response_format、`OllamaAdapter`はformat=”json”)を必須とし、LLM出力はPython側でJSON Schemaによる厳格チェックを行う。スキーマには、セリフ・感情・英語画像プロンプト・セーフティタグの4フィールドを必須項目として定義する(F-7参照)。パース失敗時は以下のフォールバックチェーン(段1〜4)を順に試行する:

- **段1 — 自動補正後の再パース:** 正規表現による自動補正(`_autofix`)、コードフェンス除去等を適用し、再パース・再検証する。
- **段2 — 補正パターン変更による再パース:** 別経路の補正パターンを適用し、再パース・再検証する。
- **段3 — プレーンテキスト形式からの抽出:** LLMがJSON非準拠のプレーンテキスト形式で応答した場合、`_try_parse_plain_format(raw)`により正規表現で`dialogue`・`image_prompt_en`・`tags`を抽出する。`emotion`は`”neutral”`を補完する。
- **段4 — 生テキストをdialogueとして使用(最終安全網):** 上記すべてが失敗した場合の最終手段として、生テキストをそのまま`dialogue`として使用する。`emotion`は`”neutral”`(固定)、`image_prompt_en`は空文字(T2I非発火)、`tags`は`[]`(セーフティタグなし)として`success=True`を返す。JSON形式で出力できないバックエンドへの最終対処であり、セーフティタグが付与されないことに注意。

段1〜3はPython側の処理でマイクロ秒オーダーであり、UX上のボトルネックとならない。旧設計で存在したLLMへの再リクエスト段(段4〜5.5)は、UXボトルネックの主因であり、カバレッジの実質的な損失がないため廃止した。既知の制約として、段3・4ではemotionが`”neutral”`固定となり、TTS・T2Iへ渡る感情情報が欠落する。

## 5.5 画像生成タイミング制御

### F-15: 柔軟なT2Iトリガー

画像生成の実行タイミングは以下の4つのモードから任意に選択可能とする。システムデフォルトは1(手動オンデマンド)、または感情の激変等のインテリジェント判定とする。

- 「手動」オンデマンド生成(デフォルト): ユーザーが生成ボタンを押した時、または特定コマンド(/image等)入力時に単発生成する。
- 「時間」インターバル自動生成: タイマーやゲーム内時間の経過に基づき自動描画する。
- 「各ターンの最後」演出先行型生成: テキストと音声が出揃ったのち、そのターンの最終結果を自動生成する(手動設定時のみ)。
- 「各ターンの最初」状況先行型生成: ターン開始直後に、現在のシチュエーションビジュアルを生成して提示する。

## 5.6 テキスト・バイナリ分離履歴管理 & session_state 軽量化

### F-16: ゾーニング・アーキテクチャ

DEF(kari) のデータ管理は、データの性質に応じた厳格なゾーニング(分離保管)を前提とする。

- **クリーンゾーン:** アプリケーションのコード、モデル特性数値マスタ(F-5、JSON)、翻訳マスタ(F-9、`locales/*.json`)、対話ログ・キャラクターシート等のメタデータ(JSON)。NSFW/ポルノ規約を完全に回避したデータのみを含み、Gitリポジトリ(クリーンゾーン管理対象)としてバージョン管理される(F-17参照)。
- **プライベートゾーン:** 生成された音声(WAV)・画像(PNG/JPEG)等の、センシティブな可能性のある生成バイナリ。ローカルの専用ディレクトリまたは外部ストレージに完全隔離保存し、Git管理対象から `.gitignore` 等で確実に除外する(F-17参照)。

F-7(セーフティタグ)・F-8(コンテンツフィルタリング)は、本ゾーニングを前提として、プライベートゾーンに格納されたバイナリへのアクセス・表示制御を行うものである。

#### キャラクターデータのファイル配置方針

キャラクターデータは1キャラクター1ファイルとし、`publish_restriction`に応じて以下のディレクトリに分けて管理する。

```
data/
  characters/
    public/       # publish_restriction: "none" のキャラクター(Git管理対象)
    private/      # publish_restriction: "private" のキャラクター(.gitignore対象)
```

`private/`ディレクトリは`.gitignore`に登録し、GitHubへの誤公開を防止する。著作権キャラクター・実在人物・個人的なキャラクターはすべて`private/`に配置する。GitHubへのpush時はF-25の公開判定ロジックにより`public/`のみが対象となる。

### F-17: テキスト・バイナリ分離履歴管理(リポジトリ肥大化対策)

セッション履歴の永続化において、Gitリポジトリが爆発的に肥大化するのを防ぐため、F-16で定義したゾーニングに基づき以下の分離管理を徹底する。

- Git管理対象: セッションログ、各種メタデータ、キャラクターシート、5.1節F-5のモデル特性数値マスタ、2.6節F-9の翻訳マスタなど、純粋なテキスト/JSONデータのみ(クリーンゾーン)。
- 外部退避対象: 生成された音声(WAV)および画像(PNG/JPEG)等の大容量バイナリファイルは、ローカルの `.gitignore` された専用アセットディレクトリ、または外部ストレージに退避保存し、JSONログ側にはその相対パスまたは識別ID(UUID)のみを記録する(プライベートゾーン)。
- Streamlit向けパス配置ロジック: 生成バイナリファイルは、Streamlitのブラウザ表示に最適化されたパス(静的ルートまたはアプリ運用中の専用ディレクトリ)へ自動配置するロジックを実装すること。
- ファイル命名規則: 歴史の分岐(Gitブランチ切り替え、5.8節F-22参照)に伴うデータの不整合・上書きを完全に回避するため、生成される音声(WAV)および画像(PNG/JPEG)のファイル名には、必ず `[キャラクター名]_[感情]_[タイムスタンプ].ext` の形式で一意の名称を付与し、プライベートゾーンへ保存する。

### F-18: session_state の軽量化

Character Persistence(1.2節)の実現により、全履歴をst.session_state上に保持することはメモリ肥大化と再描画コストの増大を招く。以下の方針で軽量化する。

- session_state には、直近の表示に必要な最小限のみを保持する。
- 完全な対話・感情・関係性の履歴は外部JSON(F-17のGit管理対象)に永続化する。
- UI再描画は、現在ターン(直近数ターン)のみを再構築対象とし、それ以前の履歴は必要時に外部JSONから遅延読み込みする。

## 5.7 長期・クロスデバイス履歴データポータビリティ

### F-19: 永続的同行データ管理(Lifelong Accompaniment Data Management)

「キャラクターと何年でも、何処にでも、一緒にいられる」という体験価値(1.5節)を技術的に保証するため、以下の機能要件を定義する。

- **フルエクスポート(Full Export):** キャラクターシート・全履歴(`mode: "chat"`の場合は`history`配列、`mode: "novel"`の場合は5.10節F-24-2の`episodes`/`chapters`/`scenes`の3階層構造)・感情履歴・関係性スコア・生成アセット参照(UUIDのみ)を単一アーカイブ(ZIP + JSON manifest)として書き出す機能を提供する。アーカイブにはフォーマットバージョンを明示し、将来バージョンでのインポート互換性を保証する。
- **インポート・マイグレーション:** エクスポートアーカイブを別デバイスまたは将来バージョンのDEF(kari)へインポートする機能を提供する。フォーマットバージョン差分に対しては自動マイグレーションスクリプトを適用する。データの安全性を検証した上で取り込み、マイグレーション失敗時はロールバックし、元データを保護する。
- **クロスデバイス同期:** ローカルファーストを前提とした同期は、ユーザーが明示的に操作した場合のみ実行する(プッシュ型)。同期経路としては(a)外部クラウドストレージへの暗号化エクスポート配置、(b)ローカルネットワーク経由の直接転送、のいずれかを選択可能とする。ユーザーの意図しないデータ通信は一切行わない。
- **データ長期保存:** 対話ログおよびキャラクターシートは人間可読なJSON形式で保存し、DEF(kari)が将来廃止される場合でもデータへのアクセスを可能とする。バイナリアセット(音声・画像)は標準フォーマット(WAV・PNG/JPEG)を維持し、独自バイナリへの変換を禁じる。

## 5.8 TRPGゲーム拡張機能 & 歴史管理

1.4節(1)で言及される「TRPGの進行を妨げない」というUX評価軸を実現するための、TRPGゲーム拡張機能群を定義する。本節の機能群(ルールブック・インジェクション、GMエージェント、ダイスロール・キャラクターシート管理)は、5.10節F-24で定義する`mode`(`"chat"`または`"novel"`)とは独立した付加機能として、いずれのモードのセッションにも有効化できる。`mode: "novel"`のセッションでGMエージェントが進行を担う場合の挙動はF-24-3で定義する。

### F-20: TRPGルールブック・インジェクション

外部のルール設定ファイル(JSON/Markdown)をセッション開始時にシステムへ読み込み、世界観やダイス(サイコロ)の判定基準をシステムプロンプトのコンテキストに組み込む。読み込まれたルールブックはF-16のクリーンゾーン(テキスト/JSONデータ)としてGit管理対象に含める。

### F-21: ゲームマスター(GM)エージェントの動的生成

F-6で登録された複数のAIキャラクターの中から1つを「GM」として指名できる。GMエージェントはF-20で読み込まれたルールブックを最優先で遵守し、進行・情景描写・ダイスロールの要求・判定を担う。GMエージェントの情景描写は、F-8のセーフティ運用ガードレールの対象として扱われ、誤った完全マスクによる進行停止を避ける。

### F-22: ダイスロール・シミュレーター & キャラクターシート管理

Streamlitバックエンド側で安全な乱数を生成し、ダイス成否結果をチャットログに自動挿入する。各AIおよび人間プレイヤーのステータスは、キャラクターシート(12章で定義するデータ構造の `game_rules_sheets`)として保持する。

- **操作ルール:** キャラクターシートのステータス増減(ダメージ計算やアイテム消費等)は、確実性を担保するため、画面(UI)からのユーザーによる手動操作・編集を基本として連動させる。
- **歴史分岐(Git操作)ルール:** システム内でのGit操作は、独立した単方向のブランチ作成(`git checkout -b`)のみとし、自動でのマージ(`git merge`)は一切行わない。これにより、分岐した歴史同士が衝突(コンフリクト)するリスクを完全に排除する。本ルールは、5.6節F-17のファイル命名規則とあわせて、ブランチ分岐時のバイナリファイル衝突を防止する。

## 5.9 ターン再生成 & Undo/Redo管理

### F-23: サイクル単位再生成 & 多段Undo/Redo管理

各ターンに対し、以下の3種の再生成操作を個別のボタンとして提供する。

- **ターン再生成:** F-14のJSON Schema構造化出力を再実行し、セリフ・感情・英語画像プロンプト・セーフティタグ(`tags`)を再生成する。再生成後の感情・セーフティタグに連動して、音声・画像も連続して再生成する(下記2項目を内部的に実行)。
- **音声のみ再生成:** F-10のTTSワーカーへ再投入する。F-11が無効(OFF)の場合は本操作の対象外とする。
- **イラストのみ再生成:** 5.5節F-15のT2Iパイプラインへ再投入する。

**Undo/Redo履歴管理:** 再生成操作を実行する都度、実行直前のターン状態(テキスト・感情・画像/音声パス・セーフティタグ等)をそのターンのUndo履歴スタックへ退避し、Redo履歴スタックをクリアする。

- **保持件数の設定:** Undo履歴の最大保持件数はユーザーがUIから設定可能とし、デフォルトは3件とする。保持件数を超えた最も古い履歴は自動的に破棄する。
- **元に戻す(Undo):** Undo履歴スタックの最新要素をポップしてターンの状態を復元し、復元前の状態をRedo履歴スタックへ積む。
- **やり直す(Redo):** Redo履歴スタックの最新要素をポップしてターンの状態を復元し、復元前の状態をUndo履歴スタックへ積む。Undo/Redoは何度でも往復可能とする。
- **永続化対象外:** Undo/Redo履歴は5.6節F-18のsession_state内のみで保持される一時的な編集履歴であり、5.7節F-19の永続化対象(外部JSON)からは除外する。新たな再生成が実行された場合、Redo履歴は破棄される。
- **エピソードモードでの適用:** `mode: "novel"`のセッションでは、本節で定義する3種の再生成操作およびUndo/Redo管理は、「ターン」の代わりに「シーン」を単位として同様に適用される。

## 5.10 エピソード生成モード

これまでのF番号群が前提としていた「1ターン=セリフ単位」のチャット形式に加え、地の文(情景・心情描写)を中心とした長編・分岐構成のノベルをLLMで生成する「エピソード生成モード」を新設する。本モードは、1.1節で言及される「Fable(物語)」の理念を実体化するものであり、チャットモードと同一のパイプライン(F-1, F-2/F-3)・リソース管理(F-12, F-13-1〜3)・セーフティ運用(F-7, F-8)・TTS(F-10, F-11)・T2I抽象インターフェース(2.3節)を共有しつつ、出力スキーマと永続化データ構造のみをモード別に切り替える。

### F-24: エピソード生成モード基盤

ユーザーはセッション開始時(または既存セッションの設定変更時)に、UIから「チャットモード」と「エピソードモード」のいずれかを選択する。モード選択結果は `mode` フィールドとしてセッション・ゲーム状態管理データ構造(12章③)に記録され、F-14(チャット用)・F-24-1(ノベル用)のいずれの構造化出力スキーマを使用するかを切り替える。チャットモードからエピソードモードへ、またその逆への変換は行わない(別モードのセッションとして新規開始する)。

### F-24-1: エピソード用構造化出力スキーマ

F-14のチャット用スキーマ(セリフ・感情・英語画像プロンプト・セーフティタグの4フィールド)とは別に、エピソードモード専用のJSON Schemaを定義し、F-14と同様にLLMバックエンドアダプター(2.3節)のJSON出力モードとPython側でのJSON Schemaバリデーション、および自動補正→再パースのフォールバックチェーンを適用する。

必須フィールドは以下とする。

- `narration`(地の文・情景描写、文字列): エピソードモードにおける主たる出力。
- `dialogue`(会話文の配列): 各要素は `speaker`(F-6で登録されたキャラクターID)・`text`(セリフ)・`emotion`(F-14と同じ感情区分)を持ち、地の文中に登場する複数キャラクターの発話を表現する。
- `tags`(セーフティタグの配列): F-7・F-8と同じ意味を持ち、地の文・会話文双方の安全性判定に使用する。
- `choices`(分岐選択肢の配列、空配列可): 各要素は `label`(選択肢の表示文字列)・`branch_id`(F-24-3で使用する分岐識別子)を持つ。空配列の場合、分岐を持たない(線形進行)。

場面挿絵の生成は、書き手が任意のタイミングでScene本文をLLMに渡し、`scene_image_prompt_en`(英語の場面挿絵プロンプト)を生成する方式とする。LLM出力スキーマには含めず、独立した生成ステップとして実装する。

エピソードモードの音声生成(F-10/F-11)は、`narration` をナレーター音声、`dialogue` 各要素を該当キャラクターの音声IDで個別に生成する。

### F-24-2: シーン・チャプター・エピソード単位の永続化データ構造

エピソードモードのデータ構造は、Episode > Chapter > Scene の3階層とする。物語は「シーン」（Scene、場面の区切り）を最小構造単位とし、複数のシーンが「チャプター」（Chapter）、複数のチャプターが「エピソード」（Episode）を構成する。ChapterとSceneの区切りは書き手がUI上のマーカー挿入ボタン（新章・新場面）で決定する。

- 作品データはタイトルごとに個別ファイル(`data/private/episodes/{タイトル}.json`)で永続化する。
- 場面挿絵・音声バイナリは、F-16/F-17のゾーニング・命名規則に従い、プライベートゾーンへ保存する。

### F-24-3: 分岐選択肢とGitブランチ連携

ユーザーがF-24-1の `choices` から選択肢を選んだ場合、F-22で定義された「歴史分岐(Git操作)ルール」(独立した単方向のブランチ作成 `git checkout -b` のみ、自動マージ禁止)に従い、選択した `branch_id` を名称に含む新しいブランチを作成してから、以降のシーン生成を継続する。これにより、選択肢によって分岐した複数の物語展開を、衝突(コンフリクト)のリスクなく並行して保持できる。GMエージェント(F-21)が存在するセッションでは、GMエージェントが選択肢提示・分岐後の進行描写を担う。

# 6. キャラクターの視覚的一貫性(Character Consistency)

1.2節で定義したCharacter Persistenceのうち、「生成される姿の連続性」を技術的に担保する仕組みを本章で定義する。

## 6.1 Consistency Provider 概念

ref_image_path は、単一の画像パスを引き渡す手段としてではなく、キャラクターの視覚的一貫性を担保するための入力源(Consistency Provider)として抽象化される。Consistency Providerは、選択されたT2Iバックエンドのアダプターによって、以下のいずれかの手法に変換・適用される。

- i2i(image-to-image): 過去の生成画像を初期画像として再生成する。
- ControlNet Reference: 参照画像の構造・特徴を保持しつつ新規生成を行う。
- IP Adapter: 参照画像の画風・キャラクター特徴をプロンプトに浸透して適用する。
- InstantID: 顔特徴を中心としたキャラクター同一性の保持に特化した手法を適用する。

どの手法を採用するかはバックエンドアダプターおよび adapter_options(2.4節)に委ねられ、DEF(kari) コアエンジン側はConsistency Providerの参照(ref_image_path)と拡張オプションを generate_image インターフェースへ渡すのみとする。

## 6.2 一貫性データの引き継ぎ(F-4・F-24との連携)

5.1節F-4で定義された動的生成において、キャラクターデータに定義されたベースイメージ、または過去のターンで生成された画像が、Consistency Providerの入力として引き継がれる。この引き継ぎ情報自体は、テキスト/JSONデータ(5.6節F-17のGit管理対象)として、画像実体とは分離して管理される。

エピソードモード(5.10節F-24)においても同様に、Scene単位の`scene_image_prompt_en`に基づく場面挿絵生成は同一のConsistency Providerを経由し、キャラクターのベースイメージおよび過去のシーンで生成された画像との視覚的一貫性を維持する。

# 7. マルチエージェント制御

5.1節F-6で定義したマルチエージェント管理基盤の下で、オーケストレーターが管理する詳細な制御ロジックを本章で定義する。本章はチャットモードにおけるオーケストレーター制御を規定する。AI卓（TRPGモード）における発言順管理・発言カウンター・キーパー制度・重要提案・投票等の自治ルールについては、『DEF AI卓 イニシアチブ・自治システム仕様 v1.0.0』を参照のこと。用語の定義は『DEF 用語集 v1.0.0』に準拠する。

## 7.1 発話優先度（イニシアチブ）

1つのSession（セッション）に登録された複数のAIキャラクターのうち、次にどのキャラクターが発話するかをオーケストレーターが決定する。チャットモードでは、直前の発話者、コンテキスト上の話題、各キャラクターのペルソナ設定上の発話傾向などに基づく優先度判定を行う。AI卓モードでは、イニシアチブ（Session開始時のランダム決定による発言順）に従い、Round単位で全参加者にTurnを巡回させる。

## 7.2 Round・Turn・Action進行制御

セッションモードにおいては、以下の階層で進行を管理する。

```
Session > Round > Turn > Action
```

- **Session**: キャラクターを読み込んでから終了までの全体
- **Round**: 全参加者に1回ずつTurnが回る単位
- **Turn**: 1キャラクターの発言機会
- **Action**: 1Turn内の個別発言（最小単位）

1Turnあたりのアクション数は設定タブで1〜5に変更可能（デフォルト3）。アクション数1の場合は従来の1発言/Turnと同じ挙動。アクション数2以上の場合、各アクションに方向性指示セットに基づく指示が付与される。

方向性指示セットは`data/public/action_directives/`に1セット1JSONファイルで管理し、設定タブのプルダウンで切り替え可能。設定は永続化される。デフォルトで以下のセットを提供する：

- **標準**（default）: 反応→体験→問いかけ→要約→本音
- **穏やか**（gentle）: 自由に語る
- **指示なし**（none）: キャラクターに完全に任せる

ユーザーはJSONファイルにエントリを追加することで独自の指示セットを作成できる。

チャットモードにおいては、AIキャラクター間の自律的な対話の往復（トス）回数に上限を設け、無制限なループによるリソース消費を防止する。最大トス回数の初期値は3回とし、ユーザーが任意に設定変更可能とする。最大トス回数に到達した場合、オーケストレーターはユーザーへ発話権を返却する。

なお、プロセス競合回避のため、AIキャラクター間の対話処理は完全なシーケンシャル（1人ずつ順番に処理）で実行される。

## 7.3 キーパー介入（Action設計）

キーパー（ユーザー）はセッション中に以下の操作でAIの発言に介入できる。

| 操作 | 内容 |
|---|---|
| **📝 送信** | キーパーアクションを積み上げる。キャラクターはまだ返答しない |
| **✅ 発言完** | 積み上げたアクションをセッション履歴に記録し、次のAI発言に指示として引き継ぐ |
| **🔄 やり直し** | 積み上げたアクションを全破棄（1ターン1回） |
| **▶ 次の発言** | アクションなしでAIに発言させる。積み上げがある場合は無効 |
| **⏹ セッション終了** | セッションを終了する |

キーパー介入はセッション履歴に🎩マーク付きで記録され、後から見返したときに「なぜここで話題が変わったか」を追跡できる。キーパー指示はプロンプト上「最優先指示」として注入される。

自動進行モード中にキーパーが介入（送信）した場合、自動進行は自動的に停止し、手動モードに切り替わる。

## 7.4 割り込み制御

ユーザーは、一時停止ボタンまたはキーパー介入による手動介入・割り込みをいつでも実行可能とする。AI卓モードにおいては、発言カウンターを消費することによる割り込み発言・次発言者指定・Turn延長も可能とする（詳細は『DEF AI卓 イニシアチブ・自治システム仕様 v1.0.0』参照）。

## 7.5 人間参加者

キャラクターデータの`player_type: "human"`を持つキャラクターがセッションに参加した場合、当該キャラクターのTurnではLLMを呼び出さず、人間プレイヤーのアクション入力を待機する。

人間参加者のTurn進行：
1. 人間参加者のTurnが回ってきたら、セッションは入力待ち状態になる
2. 人間プレイヤーは専用の入力UI（キーパー介入UIとは分離）で台詞を積み上げる
3. 以下のいずれかで確定する：
   - 「発言完」— セッション履歴に記録し、次のTurnに進行する
   - 「Turn延長」— セッション履歴に記録し、カウンターを-1消費してもう1Turn続行する
   - 「スキップ」— 発言せずに次のTurnに進行する。カウンター+1
   - 「やり直し」— 積み上げた台詞を破棄する

キーパーと人間プレイヤーのUI分離設計：将来的にAIキーパー（見えない裁定者）を導入する際、キーパー部分だけを差し替えられるよう、人間プレイヤーの参加UIとキーパー介入UIは独立して実装する。

人間参加者のキャラクターデータ（`identity_prompt`、`appearance_tags`等）は、他のAI参加者がその人間キャラクターを認識するためのコンテキストとして使用される。人間参加者自身の発言生成には使用されない。

自動進行モード中に人間参加者のTurnが回ってきた場合、自動進行は一時停止し、手動モードに切り替わる。人間参加者のTurn完了後、自動進行が再開される。

## 7.6 発言カウンター

セッション参加者ごとに発言カウンターを保持する。カウンターの初期値は0、上限は設定可能（推奨上限: 3〜5）。

### カウンターの取得（+1）

| 条件 | 値 |
|---|---|
| 自主的スキップ（自分のTurnを放棄） | +1 |
| キーパーによる強制スキップ | +1 |

### カウンターの消費（-1）

| 操作 | コスト | 効果 |
|---|---|---|
| 割り込み発言 | -2 | 発言順に関係なく発言する（スキップ+1との差し引きゼロ防止） |
| 次発言者指定 | -1 | 指名された参加者の発言終了後、元の順番に戻る |
| Turn延長 | -1 | 次のTurnも続けて発言できる |

### ペナルティ

同一発言をN回繰り返した場合、発言カウンターを-1する。Nは設定タブで可変（0=無効、デフォルト3）。エラー応答も同一発言として検出対象とする。発言カウンターがマイナスの場合、自分のTurnが回ってきても強制スキップする（強制スキップ時もカウンター+1され、自然に回復する）。

カウンターの現在値はセッションUI上のイニシアチブ順表示において、各参加者名の横に`[+1]`等の形式で表示する。現在の発言者は`▶`で強調表示する。

### カウンター操作の主体

| 操作 | 主体 | 備考 |
|---|---|---|
| 自主スキップ | プレイヤー | 人間/AI問わず |
| 強制スキップ | キーパー | 対象のカウンター+1 |
| 割り込み・次発言者指定・Turn延長 | プレイヤー | カウンターを資源として消費 |
| 投票発議 | プレイヤー（-3）/ キーパー（コスト0） | 下記7.7節参照 |

キーパーは発言カウンターを保持しない。カウンターはセッション参加者の資源である。

## 7.7 投票

セッション中にキーパーまたは人間プレイヤーが投票を発議できる。

### 投票種別

| 種別 | 効果 |
|---|---|
| お題変更 | 可決でセッションのお題を即時変更 |
| 参加者退場 | 可決で指定参加者をイニシアチブから除外 |
| セッション終了 | 可決でセッションを終了 |
| キーパー交代 | AIキーパー実装後に有効化 |

### 投票の発議

- **キーパー**: コスト0で発議可能（GM権限）
- **人間プレイヤー**: カウンター3消費で発議可能。発議者は自動的に賛成票を投じる

### 投票の処理

AI参加者はLLMで賛成/反対を自動判定（軽量呼び出し）。人間参加者はボタンで投票。過半数で可決、結果はセッション履歴に記録される。投票中は他のセッション操作をブロックする。

## 7.8 セッションルール

セッションルールは`data/public/session_rules/`に1ルールセット1JSONファイルで管理する。各ファイルは`id`、`label`、`rating`、`rules`（文字列配列）を持つ。セッションタブのプルダウンで切り替え可能。

デフォルトで以下のプリセットを提供する：

- **標準**（default）: 対立促進・対話重視
- **穏やか**（gentle）: 自由対話
- **ロールプレイ**（roleplay）: 物語重視
- **ルールなし**（none）: 完全自由

## 7.9 セッションモードTTS連携

セッションモードではAI発言生成後に同期でTTS音声を生成し、セッション履歴に`audio_path`として保持する。キャラクターごとのTTSスピーカーID（`voicevox_speaker_id`/`irodori_speaker_id`）を使用する。

再生はパイプライン方式とし、前回の音声再生中に次のLLM生成を並行実行する。前回音声の再生時間からLLM+TTS生成にかかった時間を差し引き、残り時間だけ待機してから次の発言を表示する。

人間プレイヤーの発言にTTSを適用するかはサイドバーの「あなたの音声も生成する」チェックボックスで制御する（デフォルトOFF）。

## 7.10 終了判定

オーケストレーターは、Round内の全Turn完了、ユーザーによるセッション終了操作、投票による終了決議、または会話文脈上の自然な収束（話題の終了を示すLLM出力等）のいずれかをもって、セッションの終了と判定する。

# 8. エラー制御

本章では、4章で定義した状態遷移モデルの各フェーズにおいて発生し得る障害への対処方針を定義する。すべての障害は ERROR イベント(3.2節)として共通Queueへ通知され、メインスレッドが st.session_state へ安全に反映する。

## 8.1 LLM障害

LLMバックエンド(TGW/Ollama等)プロセスの無応答・クラッシュ・JSON Schema検証の最終失敗(5.4節F-14の再試行がすべて消尽した場合)に該当する。発生時はERRORイベントを発行し、UI上にエラー表示を行うとともに、ユーザーへの再送信操作を促す。状態遷移は「LLM Processing」から「Idle」へ復帰する。

## 8.2 TTS障害

VOICEVOXプロセスの無応答または音声生成失敗に該当する。テキストは既に描画済み(4章の状態遷移上「Text Rendered」が完了済み)であるため、UX上の致命的な停止とはしない。ERRORイベントにより対応するチャットログ部分に「音声生成失敗」を示す軽微な表示のみを行い、ターン進行は「TTS Completed」相当として継続する。

## 8.3 T2I障害

選択されたStable Diffusion系バックエンドの応答失敗、またはVRAM不足によるOOMに該当する。T2I障害発生時は必ず vram_lock を解放した上でERRORイベントを発行することを徹底し、ロックの解放漏れによるデッドロック(LLM軽量レスポンスモードの恒久化)を防止する。UI上には画像生成失敗を表示し、状態遷移は「Image Running」から「Persist」へ進む(画像なしでターンを完結させる)。

## 8.4 Queue異常

イベントの取り出し順序異常、または未定義のイベント種別(3.2節未定義のtype)の検出に該当する。これらはシステム側のバグとして扱い、当該イベントをログへ記録した上で破棄し、UIの安定動作を優先する。

## 8.5 VRAM不足

vram_lock取得後もOOMが発生する場合(8.3節のT2I障害の一因)に加え、LLM・TTS・T2Iの常駐構成自体が環境のVRAM容量を超過していると判断される場合は、SYSTEM_NOTIFICATIONイベントを通じてユーザーへ警告し、F-12のリソースマネージャーによる配置(CPU/GPU分割比率、Offload設定)の見直しを促す。

# 9. メッセージ・シーケンス(イベントループ駆動安全節)

システムにおける1 サイクル（4章参照）のデータロードおよび非同期制御の順序は以下の通りである。チャットモードでは1 サイクルが1ターンの対話に、エピソードモードでは1〜N サイクルが1シーンの生成に対応する。各ステップは4章の状態遷移モデルにおける各状態に対応する。

- 入力フェーズ(Idle): ユーザーの発言、AIエージェントからの自動トス回し（チャットモード）、またはRound進行に基づくTurn開始（AI卓モード、7章参照）をメインスレッドが検知。
- LLMフェーズ(LLM Processing): 常駐しているLLMバックエンドアダプター(2.3節、デフォルト:`TgwAdapter`)に対してJSON Schema出力を強制してリクエスト。過去のコンテキストとベースプロフィールを元にしたセリフ・感情・英語の画像プロンプト・セーフティタグの4フィールドを必須項目として、JSONを確定的に一括取得する(5.4節F-14、F-7参照)。失敗時は8.1節の手順に従う。
- フロント第1描画(Text Rendered): メインスレッドは、JSONパース完了直後にまずテキスト(セリフ)のみをStreamlit上に即座に表示・描画する。この時点でユーザーに対するレスポンス待機状態を解消する。
- 非同期TTSタスク投入: テキスト描画の実行と完全に同時に、メインスレッドはTTSバックエンドアダプター(2.5節、デフォルト:`VoicevoxAdapter`)を介した音声合成処理を、専用 tts_queue(5.2節F-10)へタスク投入する。
- バックグラウンド音声生成 & キューへのプッシュ(TTS Running → TTS Completed): TTSワーカーが音声バイナリ(WAV)の生成を完了したら、完了したバイナリデータまたは保存先の相対パスを、TTS_COMPLETEイベント(3.2節)として共通Queueへプッシュする。失敗時は8.2節の手順に従う。
- 定期ポーリングによるイベント検知と第2描画: 3.4節の定期ポーリング(st_autorefresh)が走った際、メインスレッドがキューの中身をtimestamp順に確認し、TTS_COMPLETEイベントが到着していれば、安全にst.session_stateにデータを取り込み、対応するチャットログの部分に`_render_audio_player()`を通じて音声アセットを追加で反映・描画する。画面全体の強制リセットに伴わないため、音声の瞬きは発生しない。
- 待機ロック制御 & T2Iフェーズ(Image Running、トリガー実行時のみ): ユーザー要求または特定インテリジェントトリガー実行時、T2Iワーカーを起動する。
  - ① スマート・リソースマネージャー(5.3節F-12)によりvram_lockを取得する。以降、完了まではLLMへの新規要求は「軽量レスポンスモード」(5.3節F-13-2)に強制制限される。
  - ② 選択されたStable Diffusion系バックエンドのアダプターを介して generate_image を実行。F-5(プロンプト・トランスレータ)により、LLM出力の生の英語描写を選択モデルの特性数値に応じたプロンプトへ変換した上で投入する。Diffusersの場合は動的CPUオフロード(F-13-3)を有効化する。一貫性維持のため、Consistency Provider(6章)がref_image_pathとして渡される。失敗時は8.3節の手順に従う。
  - ③ 生成完了後、vram_lockを解放し、IMAGE_COMPLETEイベント(ImagePath付与)をメッセージキューへプッシュする。
- 定期ポーリングによる第3描画: 定期ポーリングがIMAGE_COMPLETEイベントを検知し、st.session_stateへ安全にマージし、Streamlit上に画像アセットを追加反映・描画する。
- 永続化 & 離脱準備(Persist → Idle): 生成された音声・画像バイナリをGit管理外のアセットディレクトリに退避保存(F-17)し、ファイル命名規則(5.6節)に従って一意な名称を付与する。メタデータと対話ログ(JSON)のみをGitでコミットし、データの一貫性のある履歴同期を完結させる。session_stateは5.6節F-18に従う直近ターンのみを保持する状態でIdleへ復帰する。

# 10. 将来拡張方針

コアエンジンをUIから分離し、FastAPI/WebSocket/React/Electronへの移行可能性を維持する。2.1節で述べた通り、現段階の実装はStreamlitを基盤とするが、以下の境界を保つことで将来的な移行コストを最小化する。

- コアロジック(イベント処理・状態遷移モデル・各AIレイヤーとのインターフェース)はUIフレームワークに依存しない形で実装し、Streamlit固有のAPI(st.session_state、st_autorefresh等)への依存をUI層に限定する。
- 3章で定義した共通Eventスキーマは、将来的にWebSocketを介したサーバー・クライアント間通信のメッセージ形式として再利用可能な構造とする。
- 2.3節のT2I抽象インターフェース、および5.2節のTTSワーカーモデルは、FastAPI移行後に独立したサービスとして切り出せる粒度で設計する。

## ② キャラクター(エージェント)ポータブルデータ構造(F-4, F-6, F-20〜F-22)

### データ管理方式

キャラクターデータは1キャラクター1ディレクトリで管理する。各ディレクトリに`profile.json`と画像ファイルを配置する。公開可能なキャラクター（`rating_sexual`が`general`/`sfw`かつ`rating_violence`が`general`/`violence`）は`data/public/characters/`に、NSFW以上のキャラクターは`data/private/characters/`に配置する。`data/private/`はGit管理対象外とし、開発ツール（Claude Code等）からの読み取りリスクを回避する。

```
data/public/characters/              # 公開キャラクター（general/sfw）
  character_claude_001/
    profile.json
    icon.png
    standing.png

data/private/characters/             # NSFWキャラクター（git除外）
  character_xxx_001/
    profile.json
    icon.png
    standing.png
```

アプリケーションは両ディレクトリを読み込み、ディレクトリ名をキャラクターIDとしてプルダウンに統合表示する。

### キャラクター画像

キャラクターごとにアイコンと立ち絵を保持する。画像は`public/characters/{character_id}/`または`private/characters/{character_id}/`ディレクトリに以下の規約で配置する。

| 種別 | ファイル名 | サイズ | 用途 |
|---|---|---|---|
| アイコン | `icon.png` | 512×512 | チャット・セッションのアバター表示 |
| 立ち絵 | `standing.png` | 832×1216 | セッションモードの背景表示 |

画像はキャラクタータブから取り込み（ファイルアップロード、自動リサイズ）またはT2Iバックエンドで生成可能。ファイルが存在すれば表示、なければデフォルトアイコン（絵文字）でフォールバックする。キャラクターデータJSONへの画像パス記録は行わず、ディレクトリ規約で管理する。

```json
      "character_copilot_001": "信頼できる編集者。控えめだが正確で、私の書いた文章を構造的に整えてくれる。"

    "game_rules_sheets": {

```

- **`name_reading`(名前の読みと別名)**: VOICEVOXへの読み渡し・UI表示に使う名前情報を管理するオブジェクト。
  - `family_name`(姓の読み、カタカナ): 姓がない場合は空文字。
  - `given_name`(名の読み、カタカナ): 必須。
  - `alias`(別名リスト): 源氏名・芸名・愛称・ペンネーム等を配列で管理する。各要素は`{"name": string, "reading": string | null}`。`reading`は漢字を含む場合に設定し、不要な場合は`null`。空配列も可。
- **`identity_prompt`(キャラクターの本質・性格)**: LLMのシステムプロンプトに常に組み込まれるテキスト。服装(`outfits`)・容姿(`appearance_description`)・口調(`speech_style`)等の専用フィールドに切り出せる情報は含めず、キャラクターが「どう在るか」（内面・性質）のみを簡潔に記述する。「何ができるか」（能力・スペック）は`identity_detail`に記述すること。必須項目。
- **`identity_detail`(設定の補足)**: `identity_prompt`に収まらない詳細な設定情報（能力・仕様・背景・経緯・癖・趣味等）を記述する任意項目。「何ができるか」はここに記述し、`identity_prompt`には含めない。設定がある場合はLLMのシステムプロンプトに`identity_prompt`の後に追加する。`null`または省略した場合は展開しない。
- **`player_type`(操作主体)**: `"ai"` | `"human"`。デフォルトは`"ai"`。`"human"`の場合、セッションモードにおいて当該キャラクターのターンではLLMを呼び出さず、人間プレイヤーのアクション入力を待機する。`"ai"`の場合は`default_model_config`に基づいてLLMが発言を生成する。チャットモードでは本フィールドは参照されない。`persona_attributes`ではなく`base_profile`直下のフィールドとして定義する。
- **`persona_attributes`の各フィールド:**
  - `gender`(性別)・`gender_identity`(性的自覚、身体的性別とは別に本人が自覚している性別): いずれも `"男" | "女" | "その他"` のいずれか。LLMのシステムプロンプトに自動展開する。`gender_identity`は`gender`と異なる場合のみ展開。
  - `romantic_interest`(恋愛対象): `"男" | "女" | "その他"` の配列(複数選択可)。空配列の場合は「恋愛対象: なし」としてLLMに展開。`rating_sexual`の値にかかわらず常にLLMに渡す（キャラクターの身体性と感情の深層として人間味に影響するため）。
  - `actual_age`(実年齢)・`appearance_age`(外見年齢): 数値。キャラクターの設定上の年齢と、見た目年齢を分離して保持する。
  - `appearance_description`(容姿の詳細な説明文): 体型・髪・目・顔立ち等、**変化しない容姿のみ**を記述する。服装は含めない。任意項目で、未設定の場合は`null`または省略する。
  - `past_life`(前世情報): 転生キャラクター専用の任意項目。前世を持たないキャラクターは`null`または省略する。`raised_in`は転生後の情報のみを記述し、前世の環境はこのフィールドで管理する。フィールド: `origin`(前世の属性・立場、文字列)・`cause_of_reincarnation`(転生の経緯、文字列、任意)。
  - `roles`(役割・職業リスト): キャラクターの職業・役割を配列で管理する。複数の役割を持つ場合はすべて列挙する(例: `["歩き巫女", "間者"]`)。`identity_prompt`に職業を重複して記述しないこと。空配列も可。
  - `primary_role`(主役割): `roles`の中でキャラクターを最もよく表す主な職業・役割を文字列で指定する。LLMのシステムプロンプト展開・T2Iプロンプト生成時に優先使用する。`roles`が空の場合は`null`。
  - `outfits`(衣装辞書): キャラクターの衣装を辞書形式で管理する。キー名(`"default"`, `"casual"`, `"battle"`等)で衣装を識別し、値に服装の説明文を持つ。`"default"`キーは必須。セッション中の衣装変化は`session_state`側の`current_outfit`フィールド(キー名を文字列で保持)で管理し、`current_outfit`が`null`または未指定の場合は`"default"`にフォールバックする。T2Iプロンプト生成時は`outfits[current_outfit]`の値を服装情報として展開する。
  - `era_presets`(時代設定プリセット辞書): 歴史上の人物や時代設定のあるキャラクターの時代・年代・場所・その時代での年齢を辞書形式で管理する。`outfits`と同じ構造で、`"default"`キーは必須。現代キャラクターは`null`または省略する。セッション中の時代切り替えは`session_state`側の`current_era`フィールド(キー名を文字列で保持)で管理し、`null`または未指定の場合は`"default"`にフォールバックする。各プリセットのフィールド: `period`(時代名称、必須)・`year_range`(年代範囲、任意)・`location`(場所、任意)・`era_age`(その時代での年齢、任意。設定した場合は`actual_age`より優先してLLMへ展開する)。
  - `speech_style`(一人称・相手の呼び方・口調等を持つオブジェクト): 任意項目で、未設定の場合は`null`または省略する。
  - `cultural_background`(文化的背景): キャラクターの価値観・言語感覚・行動様式の形成に関わる背景情報を3フィールドで保持するオブジェクト。任意項目で、未設定の場合は`null`または省略する。
    - `birthplace`(生まれた場所): 出生地の記録。キャラクター設定資料としての情報であり、LLMのシステムプロンプトへの展開は行わない。
    - `raised_in`(育った場所・期間): 価値観・言語・行動様式を形成した環境。「ニューヨーク(10歳〜18歳)」のように期間を含めた自由記述も可。複数の場所で育った場合は配列とする。LLMのシステムプロンプトへ展開し、ペルソナ・口調の形成に最も直接的に影響する。
    - `dominant_culture`(支配的な文化圏): 育った場所だけでは表現しきれない、キャラクターのアイデンティティの核となる文化的帰属。`raised_in`が「どこで」かを表すのに対し、「どの文化に最も強く影響されているか」という抽象的な属性を示す。LLMのシステムプロンプトへ展開する。
- **`content_policy`の各フィールド:** F-25参照。キャラクターデータのGitHub公開可否を判定するためのフィールド群。`is_real_person: true`のキャラクターは著作権消滅の有無にかかわらず`publish_restriction: "private"`固定とする。
- 1セッションに複数登録可能なAIキャラクター(F-6)は、本データ構造のエントリ(`character_claude_001`等)を複数持つことで表現する。キャラクター切替機能は、本構造の複数エントリから対話相手として1キャラクターを選択するUIに対応する。
- **`default_model_config`の各フィールド:**
  - `text_model_id`: F-5モデル特性数値マスタのLLMエントリID。
  - `image_model_id`: F-5モデル特性数値マスタのT2Iエントリ(MVPではA1111バックエンド固定)のID。
  - `voicevox_speaker_id`: VOICEVOXのスタイルID(整数)。`VoicevoxAdapter`使用時に参照する。
  - `gemini_tts_voice`: Google AI Studio Gemini TTSの音声名(文字列、例: `"Aoede"`)。`GeminiTtsAdapter`使用時に参照する。VOICEVOXの整数型話者IDとは別管理であり、アダプター切替時に対応するフィールドを使い分ける(2.5節参照)。
  - `irodori_speaker_id`: Irodori-TTSの参照音声ファイル名(文字列、`data/irodori_speakers/`配下、例: `"luna_ref.wav"`)。`IrodoriTtsAdapter`使用時に参照する。空文字列の場合は参照音声なし(ランダムボイス)として合成する。`voicevox_speaker_id`・`gemini_tts_voice`とは別管理であり、アダプター切替時に対応するフィールドを使い分ける(2.5節参照)。
  - `location`: 推論実行場所(`"local"` または `"remote"`)。

## ③ セッション・ゲーム状態管理データ構造(F-7, F-14, F-20〜F-22, F-17, F-18, F-9, F-24)

`mode` フィールドは F-24 で定義したモード選択(`"chat"` または `"novel"`)を示す。`mode: "chat"` の場合は本節の `history` 配列を、`mode: "novel"` の場合は12章④で定義する `episodes` 配列を用いる(両方を同時に持つことはない)。`history` 配列の各要素は、5.4節F-14のJSON Schemaで必須化された `emotion`(感情)・`tags`(セーフティタグ、F-7)を含む。`image_path`/`audio_path`は、5.6節F-17の命名規則に従ったプライベートゾーン内の相対パスを記録する。`language`フィールドはF-9の言語分離における表示言語設定を示す。

```json
{
  "session_id": "def-trpg-session-001",
  "mode": "chat",
  "language": "ja",
  "active_rule_system": "trpg_coc_style",
  "game_system": {
    "rulebook_summary": "簡易版TRPGルール。ダイスは1d100を使用。"
  },
  "history": [
    {
      "sender": "character_claude_001",
      "text": "ちょっと、勝手に前に出ないでよ！",
      "emotion": "angry",
      "image_path": "private_zone/history/luna_angry_20260611_160400.png",
      "audio_path": "private_zone/history/luna_angry_20260611_160400.wav",
      "tags": ["mild_violence"]
    }
  ]
}
```

`session_state`(F-18)には、本データ構造のうち直近の表示に必要な最小限(直近数ターンの`history`要素)のみを保持し、全履歴は本構造の永続化JSONファイルとしてF-17のGit管理対象に保存する。

## ④ エピソードモード章構成データ構造(F-24)

`mode: "novel"` のセッションは、12章③の `history` 配列の代わりに `episodes` 配列を持つ。物語は「シーン」(最小構造単位)・「チャプター」(シーンの集合)・「エピソード」(チャプターの集合)の3階層で構成される。各シーンは5.10節F-24-1のスキーマで定義された `narration`・`dialogue`・`tags`・`choices` を含む。`branch_id` は、F-24-3で定義した分岐選択時のGitブランチ名と対応する。`narration_audio_path` はナレーター音声、`dialogue` 各要素の `audio_path` はキャラクター個別の音声を示す。場面挿絵は `scene_image_prompt_en` に基づきScene単位で生成する。

```json
{
  "session_id": "def-novel-session-001",
  "mode": "novel",
  "language": "ja",
  "episodes": [
    {
      "episode_id": "ep01",
      "title": "第一部 出会い",
      "chapters": [
        {
          "chapter_id": "ep01_ch01",

              "scene_id": "ep01_ch01_sc01",
              "title": "謝罪",

              "narration_audio_path": "private_zone/novel/narrator_neutral_20260612_170000.wav",
              "dialogue": [
                {
                  "speaker": "character_claude_001",
```
